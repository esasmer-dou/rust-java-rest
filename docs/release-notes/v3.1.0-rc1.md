# v3.1.0-rc1 - Rust I/O Plane Performance Preview

This is a release candidate, not a stable production claim. The product direction is now explicit:
Java owns business logic and application composition; Rust owns the HTTP I/O plane, bounded native
memory, selected serialization-heavy paths, large/static response handling, and backpressure.

## Positioning

BEST use cases:
- Small JSON APIs where Java handlers should avoid Spring Boot runtime overhead.
- Read-heavy or precomputed/raw JSON responses.
- Large file/static/export responses served by Rust without moving body bytes through Java.
- Services that need bounded RSS and predictable overload behavior.

ACCEPTABLE use cases:
- Dynamic JSON endpoints where business logic remains Java and response serialization uses direct
  writer/generated-style paths.
- WebSocket workloads with bounded outbound queues and explicit slow-consumer handling.

ANTI-PATTERN:
- Claiming universal "5x faster than Spring Boot" for every endpoint.
- Building large Java DTO object graphs per request and expecting Rust I/O alone to remove that cost.
- Increasing request/response limits without in-flight byte caps.

## Key Changes

- Added bounded JNI worker queue and explicit backpressure behavior.
- Added configurable request body, response body, in-flight byte, header, timeout, and keep-alive limits.
- Added response buffer pooling, response body ownership improvements, and native memory diagnostics.
- Added `RawResponse`, `FileResponse`, static/native response registration, and large response paths.
- Added WebSocket outbound send registry with bounded queues and path/query support.
- Added low-overhead metrics and hot-path logging cleanup.
- Added low-RSS, ultra-low-RSS, micro-RSS, and throughput runtime benchmark profiles.
- Added direct JSON writer APIs and generated-style writer/parser prototypes for hot DTO paths.
- Added `@DirectQueryInt` for primitive query parameter passing from Rust to Java without Java query
  `String` allocation on selected direct routes.
- Rebuilt Windows `rust_hyper.dll` and Linux `librust_hyper.so` for native ABI 9.

## Production Gate

This RC can be used for preview and controlled benchmark validation. Before stable release:

- Run benchmark with `RepeatCount >= 3` and publish median/p95 across runs.
- Tune JNI queue/backpressure to reduce or eliminate low-rate `503` at c=1000.
- Restore CI/CD matrix for Windows DLL and Linux glibc `.so` builds.
- Align Rust native source repository access with Java framework release artifacts.
- Document `RawResponse`, `FileResponse`, `@DirectQueryInt`, runtime profiles, metrics, and limits.
